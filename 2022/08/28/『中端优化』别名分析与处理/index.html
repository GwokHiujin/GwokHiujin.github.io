『编译优化』Alias Analysis: 别名分析与处理

正如我在前面的博文里废话过多次的，别名分析没时间做是我打编译比赛的一大遗憾，搞定了它以后值编号的强度将上升一个台阶，同时也能为更多的优化暴露时机。在很多关于全局优化的文献里，一般都假定已经执行过了别名分析并且不再提及它，因此读者无论如何必须知道，别名分析是正确且高效地实现大多数优化的保证。

一开始愚钝的我没意识到这一点，因此引发了很多的段错误——原因是 load store 指令不能轻易直接替换，应该要做一个别名分析，将 load 指令与某个 store 指令进行绑定，判定指令相同时，除了判断操作数，还要判断这个 store。因为这两个指令操作的是地址，对于两个 load 指令，即使它们长得一模一样，从该处取出的值也可能完全不同（因为从同一个地址来，两次 load 中间可能发生过针对该地址的 store）。等我意识到这一点的时候我手上的优化工作量还挺大，我们又人手不足，所以最后的 GVN 舍弃了对 load 和 store 的处理，是跛足前行的编译器一枚呀（泪）

背景

高质量的别名信息对激进的和正确的优化是重要的。如果两个或多个表达式表示相同的内存地址，我们称这些表达式互为别名（Alias）。如果以 LLVM 为例的话，最显著的想来就是 load, store 和 GetElementPtr 三板斧了，基于它们完成的别名分析优化被称为数组 SSA；如果以 C 的变量为例，显然我们可以取其地址，并且可以通过变量名或者指针来读写它。

事实上，指针和过程都有可能引入别名，本文也是讨论包含指针和过程的数据流分析。对于大多数 C 程序而言，我们只需要假定：只有那些取了其地址的变量会发生别名，并且任何指针值变量都可能指向它们。多数情况下这种假定对优化的限制最小。但是针对一些特别的例子（比如一个有 200 个变量的程序，100 个变量取了其地址，另外 100 个是指针）就需要采取更加激进的别名分析。

别名分析有许多不同的算法和许多不同的分类方法：流敏感 vs 流不敏感，上下文敏感 vs 上下文不敏感，字段敏感 vs 字段不敏感，等等。传统上，别名分析输出 Must、May 或 No Alias 的结果响应别名查询，分别表明两个指针总是指向相同的对象、可能指向相同的对象、或已知永远不会指向相同的对象。根据精确度的不同，我认为把别名分析分成下图中所列的 4 种（注意是元素两两拼起来有 4 种）是比较合适的，不同精确度的工作量和效果都不同。其中 x alias y 的释义是“x 指向 y”，我们不关心这个关系是否是自反的或者反自反的，因为 x alias x 没有意义。

这个分类在 LLVM 的别名分析文件中也有体现，它输出的别名分析结果就是前面提到的 NoAlias, MayAlias, PartialAlias 和 MustAlias。（代码来自 AliasAnalysis.h）

产生别名的原因因编程语言而异，但各种语言的别名计算具有共性成分。因此，我们将别名计算分为两部分：

  1. 别名收集器（Alias Gatherer）：语言特有的部分，由编译器前端提供；
  2. 别名传播器（Alias Propagator）：优化器中的一个单独部分，利用前端发现的别名关系执行数据流分析，组合别名信息并将它们传递到需要它们的程序点去

语言特有的别名收集器应当可以发现由于以下原因发生的别名：

  1. 为两个对象分配了重叠的存储单元
  2. 数组、数组片段或数组元素的引用
  3. 指针引用
  4. 参数传递
  5. 上面各种机制的组合

别名收集的过程，是使用一些方法将潜在的别名对象映射到抽象存储位置上去。在这里我们强调一个收集别名时的准则：如果在两个对象之间可能存在别名关系，而又不能证明这个关系不存在，则我们必须记录它可能存在，以免由于遗漏了实际存在的别名而导致结果的语义错误。也就是说，我们最终得到的是一个变量可能指向的内存的集合。

收集别名的时候，作为编译器编写者的我们可以对自己的收集选项做很多选择，比如区分可能存在的别名的精细度、是否区分动态分配的存储区域和其他存储区域、选择流敏感的还是流不敏感的别名分析……等等，做区分一般需要更多的编译空间和编译时间，但可能产生更好的代码，我们应当在其中做一些自己的 tradeoff，做什么样的选择既与我们的精力有关也与我们的工作经验有关。综上所述，指针使用的灵活性导致了别名分析的复杂性，导致实际上对指针信息的分析是一个不可判定问题，所以实际的指针分析算法都是近似且保守的，须在效率和精度之间进行折衷。

下面，我们讨论别名分析的两个经典算法：Andersen 算法和 Steensgaard 算法。它们都是流不敏感的，并且实际上是指针分析算法，别名分析的功能不过是其子集。因为别名分析讨论的是变量对 (p, q) 中 p 和 q 是否指向同一个地址，而指针分析还可以讨论变量对 (p, x) 中 p 是否指向 x 的地址。可见，指针分析应当能暴露更多的优化时机。

指针分析研究的内容主要集中在分析精度和时空开销之间的取舍，精度方面，主要指流敏感性（flow-sensitivity）和上下文敏感性（context-sensitivity），一般而言，流敏感分析方法的精度明显好于流不敏感的分析方法，在上下文敏感性上也有同样的特点，但出于经验主义的取舍，主流编译器反而更多采用流不敏感的分析方法。

流不敏感的指针分析普遍使用在开源或者产品级高级编译器中，其中主要有两类：基于包含（inclusion-based）的指针分析和基于合并（unification-based）的指针分析。基于包含的指针分析是一种基于约束集（constraint set）求解的流不敏感的指针分析方法，该指针分析又称为基于子集（subset-based）的指针分析或者基于约束的（constraint-based）的指针分析，我们要讨论的 Andersen 和 Steensgaard 算法都属于这种类型的指针分析算法。

顺带一提，LLVM 的别名分析类采取的是 Steensgaard 算法。

Andersen’s Points-To Analysis

Andersen 算法是一种基于约束（constraint-based）或者说基于子集（subset-based）的方法，根据程序中的语句建立变量与变量，或者变量与内存之间的约束关系。显然，要完成别名分析的目的的话，只要判断两个变量之间是否存在相等的约束关系就可以了。Andersen的方法分为两步：约束生成（constraint generation）和约束求解（constraint resolution）。

从上一节的内容中我们也已经谈到，Andersen 算法讨论的是一种上下文不敏感的过程间分析，也是一种流不敏感分析，这种分析（与数据流分析不同）不考虑程序语句的顺序，只管那些赋值的、和指针相关的语句，而不管关于控制流的那些东西，特点是很安全，并且成本远远低于精确的指针分析，但是显然损失了精确性。

约束生成

此算法感兴趣的是分析程序中与指针相关的指令。忽略内存分配和数组，我们可以将所有指针操作分解为四种指令类型：获取变量的地址，将指针从一个变量复制到另一个变量，通过指针赋值，和对指针解引用。基于此，我们将约束类型分为基本约束、简单约束和复杂约束三种，并且有如下的具体形式：

Constraint Type Assignment Constraint Meaning
Base Referencing（a := &b） a ⊇ {b} loc(b) ∈ pts(a)
Simple Aliasing（a := b） a ⊇ b pts(a) ⊇ pts(b)
Complex Dereferencing read（a := *b） a ⊇ *b ∀ v ∈ pts(b), s.t. pts(a) ⊇ pts(v)
Complex Dereferencing write（*a := b） *a ⊇ b ∀ v ∈ pts(a), s.t. pts(v) ⊇ pts(b)

其中，loc(b) 表示 b 的地址（location b），pts(a) 表示 a 可能指向的内存的集合。显然，如果 pts(a) 中只有一个元素，那么 a 所包含的就是一个 must-point-to information，否则是一个 may-point-to information。值得注意的是，如果 a 中只包含一个 malloc 描述或者 new 描述，我们依旧将其处理为一个 may-point-to information（即 a 只指向在该程序位置被分配的许多内存单元中的一个），因为我们将这两个语句分配的内存单元视为一种抽象的位置，它们的内存分配操作完全可以被多次执行。

如果是在 LLVM 的情形的话，约束生成的过程是很简单的，因为 LLVM 有清晰的指针类型指令，还可以方便地查询指令操作数的值类型，以供我们鉴别此时的约束类型。

约束图

介绍约束求解过程之前要先引入约束图的概念。对于一个约束图 ，图中的节点 Variable 为变量或者抽象的内存区域，每一个节点都关联一个对应的 point-to Sets（即图中的边）。对于一条 a->b（这里表示的是约束关系而不是指向关系） 的边 e，e ∈ E 当且仅当如下三种情况之一成立：

  • pts( a ) ⊆ pts( b ) ：{ 简单约束 }
  • a ∈ pts( v ) && pts( *v ) ⊆ pts( b ) ： { 复杂约束 }
  • pts( a ) ⊆ pts( *v ) && b ∈ pts( v )：{ 复杂约束 }

初始化约束图需要先收集基本约束和简单约束的约束信息，按以下三步进行初始化：

  1. 为程序中的每个变量建立一个节点
  2. 根据基本约束标注节点的指向集（只要有指针类型的指令就很好解决这一点了）
  3. 为每一个初始的简单约束建立一条有向边

往约束图中加边的方法分为两种：

  1. 生成约束信息时，根据基本约束直接添加边
  2. 进行约束求解时，根据复杂约束进行约束关系的推导，再添加边

第二种加边的情形事实上就是约束求解。需要阐明的是，我们的最终目的不是获得约束图，而是在创建约束图的过程中将 point-to information 进行正确的传播。这些 point-to information 按照下面的原则进行传播：

Steensgaard’s Points-To Analysis

Steensgaard 算法也是流不敏感的，主要是在 Andersen 算法的基础上简化了问题，加快指针分析的速度，但比之又损失了一些分析精确度。它跟 Andersen 算法的区别在于简化了约束条件：

Constraint Type Assignment Constraint Meaning
Base Referencing（a := &b） a ⊇ {b} loc(b) ∈ pts(a)
Simple Aliasing（a := b） a = b pts(a) = pts(b)
Complex Dereferencing read（a := *b） a = *b ∀ v ∈ pts(b), s.t. pts(a) = pts(v)
Complex Dereferencing write（*a := b） *a = b ∀ v ∈ pts(a), s.t. pts(v) = pts(b)

可以看到，就是把子集关系直接换成了等号，大大简化了指针分析，因而算法复杂度可以达到 。从对约束条件的修改可以看到，Steensgaard 用以加快分析速度的底层逻辑是对程序中的每个变量使用常数空间。他的分析将每个变量 p 与一个以该变量命名的抽象地址联系起来，然后，跟踪 p 和它可能指向的另一个位置 q 之间的单一 point-to 关系，也就是说，原来是点对集合的映射，现在简化成点对点的映射。当然，在实际的程序中，p 完全有可能同时指向 q 和其他变量 r （甚至更多），在这种情况下，Steensgaard 的算法将 q 和 r 的抽象地址统一起来，创建一个单独的抽象地址来表示它们。例如，考虑下面的程序:

流敏感的别名分析

前面介绍的两种经典算法都是流不敏感（即不考虑程序语句执行的顺序）的，这样会牺牲分析的精度。流不敏感的别名分析是如今的编译器主流（因为对于大型项目程序来说，其 CFG 是很庞大的，节点也很多，因此以迭代式数据流分析为框架的流敏感分析会非常低效，一个解决办法是采用稀疏分析跳过无用节点，优化流敏感信息流的传递过程，但这就涉及 另一篇论文 了，我放在博文末尾），而流敏感的指针分析一般用于分析多线程的情形。

显然，在这里我们需要利用迭代数据流分析技术来收集别名信息，并将别名信息从它们的定义点传播到过程中所有可能需要它们的地方。流敏感方法为每个程序点都计算了该点独有的程序状态。流敏感别名分析使用的流图结构是用控制流边连接的各个语句组成的，转移到 LLVM 中就是我们计算生成的支配树结构（我假设读者已经会了，不会的话可以查找任意一本教科书中有关迭代数据流分析的部分）。然后我们需要顺序遍历支配树，为其赋予别名概念。

其具体算法内容建议直接参照鲸书（ Advanced Compiler Design and Implementation. Steven S.Muchnick ）的第 10 章，我就不想抄书了（逃）个人的感觉是这一分析过程实在不太优雅，也比较繁琐，主流编译器想来也是在其复杂度与性能之间做了合适的 tradeoff 的。

使用别名分析结果

进行别名分析（实际上是指针分析）后我们得到了两个指针之间 May、Must 或者 Not Alias 的结果，接下来看如何使用。此处是以 LLVM 的 AliasAnalysis 类为例，针对你自己写的别名分析也是同理。

MemoryDependenceAnalysis Pass

memdep pass 使用别名分析来提供内存使用指令的高级依赖信息，例如，告诉你哪个 store 提供给了哪个 load，只需要查找 load、store 指令存在的别名信息对（Pair）就可以了，当然，也可以分析得到别名信息对之后将对应的 store 信息作为一个属性存到 load 指令类中（利用面向对象语言的优点）。它也使用缓存等技术提高效率，一般用于 Dead Store Elimination（删除别名信息对为空的 store 指令）, GVN（比较 load 指令对应的 store 别名，如果相同则可以替换）和 memcpy 优化。

AliasSetTracker

有些代码转换需要某个代码区域内的活跃的别名集，而不是成对的别名信息。此时可以建立 AliasSetTracker 类，根据 AliasAnalysis 的结果提供的成对的别名分析信息，高效地构建所需的别名集。

直接使用 AliasAnalysis 接口

如果这些功能类都用不到，你可以直接使用 AliasAnalysis 类服务自己编写的某些优化算法。此处应尽量使用高级方法，以获取更高的准确率和效率。
