『编译优化』Strength Reduction: 运算符强度削减

强度削减（Strength reduction）是一种常见于 循环 的优化，属于特化计算的优化范畴，功能顾名思义是减小程序的“运算强度”。它的本质实际上是很多编译技术书籍中会为之单开一章的归纳变量及其优化。

我找了一会儿资料后并没有发现针对“运算强度”这个名词的显式定义，从我对这一优化的浅显了解来看，减小运算强度的过程是 以低成本操作取代高成本操作 的过程，比如一条乘法指令的成本显然高于一条加法操作，而超过一定数量的一系列加法操作的成本又将高于一条乘法操作。如有更详细的定义，欢迎在评论区中补充。

背景

强度削减会考察一个操作所处的上下文环境，且基于强度削减优化的动机，这个操作应当在循环内部执行，其操作数分别是：

  1. 区域常量（Region Constant），一个在循环中不改变的值
  2. 归纳变量（Induction Variable），一个随循环的各次迭代发生系统化改变（按某个常量递增或递降）的值，比如经典的计数变量 i

算法在发现这种情况时，会创建一个新的归纳变量，以一种更高效的方式来计算出一个与原来的操作结果等价的序列，一般来说，这种算法适用于乘法操作。对于一个乘法操作，如果其操作数符合上面的形式约束，那么就可以确保新的归纳变量能够利用加法而非乘法来计算，在大多数情况下这是可以减小程序运算强度的。

区域常量

区域常量可以是一个字面意义上的常数，也可以是一个循环不变量（即在循环内部不修改的值）。利用 SSA 形式下的代码我们可以比较轻松地检索循环不变量，只要检查操作参数的定义位置是否唯一就可以了，当然，这个定义必须支配定义归纳变量的循环的入口程序块。如果在强度削减之前执行代码提升优化（此优化完成了循环不变量外提的操作）和常量传播可能可以揭示更多的区域常量，因此需要把强度削减放在它们后面执行（时刻注意考虑优化序列的问题）。

归纳变量

识别归纳变量是一个值得讨论的专题。为了有利于识别，常常将归纳变量分为基本归纳变量和依赖归纳变量两类，前者是在循环的每一次迭代中显式地加或减一个相同常量的变量，而后者的修改或计算则以一种较为复杂的方式进行。比如以下这段 Fortran 77 归纳变量的例子：

因此，寻找候选操作的过程，就是寻找依赖归纳变量的过程。因而，强度削减算法的一个更常见的说法是“归纳变量强度削弱”。我以这种顺序引入算法背景，主要是因为几本经典的大部头里的说法都不太统一，这里稍微把它们杂糅一下，这样再去看任意一本书的时候就不太容易混淆一些基本概念了。

后面我将采用鲸书的说法，即基本归纳变量和依赖归纳变量，对强度削减算法进行阐述。

归纳变量删除

除了对归纳变量进行强度削弱外，我们常常还能完全删除它们。删除它们的基本判断标准是显然的，即这种归纳变量在程序中没有用途，它们的产生原因通常如下：

  1. 这种变量从一开始就没有参与计算；
  2. 由于其他优化转换导致这种变量变为无用的，比如上面的强度削弱；
  3. 这种变量是在一次强度削弱的过程中临时创建的，然后因为进行了另一次强度削弱而变为了无用的；
  4. 这种变量只用在循环结束测试中，并且可以被那个上下文中其他的归纳变量所替代

针对大部分的情形，我们可以简单地用死代码删除来完成；针对第四种情形，其对应的归纳变量删除算法在我看来相当恶心……它的底层逻辑是“线性函数测试替换”，即对每个依赖归纳变量 j，如果它本身的 b 和 c 比较简单（比如 b = 1 且 c = 0），就尝试用 j 去代替 i，观察是否能被代替。基本上在比较复杂的情况下测试的替换都是缺少价值的，不是特别想细讲……个人感觉引入一个死代码删除在这里意思意思一下就好了。

  • 线性函数测试替换

强度削减结合死代码删除通常可以消除一个归纳变量的所有使用，但循环结束处的条件判断除外（在经典的死代码删除算法里，跳转指令会被固定为有效指令，不可删除）。在这种情况下，编译器也许能够重写循环结束处的条件判断来使用循环中发现的另一个归纳变量，如果编译器能够删除这最后的一处引用，它就可以将原来的归纳变量作为死代码删除。这一变换过程就是线性函数测试替换（LFTR）。

讨论一条这样的三地址指令：if i relop x goto n（如果 i 和 x 满足 relop 关系，就跳转到 n 对应的指令），其中 x 不是归纳变量。测试替换的过程，就是现在我们在 i 类中取某个 j，优先取其三元组（i, b, c）中的 b 和 c 尽可能简单的 j（即优先考虑 b = 1 和 c = 0 的情形），把每个含 i 的测试改成用 j 代替 i，检查条件跳转的结果是否正确。

针对上面的 j，显然现在可以构造一条这样的测试：

r := b * x

r := r + c

if j relop r goto n

其中 r 是新构建的临时变量。对于 if x relop i goto n 的处理与此类似。如果结果是正确的，就用这条语句去代替原来的三地址指令，以期暴露更多可优化的归纳变量。

如果现在的测试语句中 i 和 x 都是归纳变量，那么就要检查 i 和 x 是否都能被代替。最简单的情况是 i 和 x 对应的类中都有依赖归纳变量供我们测试，但在更复杂的情况下，测试的替换就可能是没有价值的。

还有一种值得考虑的情形是检查归纳变量强度削弱时为其引入了语句 j := s 的归纳变量 j，检查在引入的 j := s 和任何对 j 的引用之间有没有对 s 的赋值，如果没有，就用对 s 的引用代替所有对 j 的引用，并删除 j := s 语句。
