『编译优化』Local Value Numbering: 局部值编号

这几天的生产 bgm 是来自美国洛杉矶的前卫金属乐队 Mammoth (Thrailkill) 的专辑 Deviations，非常耐听，尤其是 The Hilarity of Singularity 这首中的融合爵士风格十分有意思。

作用于局部范围即单个基本程序块上的优化，是编译器能够使用的最简单的技术之一，最著名的两种局部方法是：值编号（value numbering）和树高平衡（tree-height balancing）。前者用于查找基本程序块中的 冗余 表达式，通过重用此前计算过的值来替换冗余的求值；后者用于重新组织表达式树，以揭示更多指令层次的并行性。出于本人的工作重点与工期问题，在这里我暂时只关注局部值编号的问题。

冗余：如果在通向位置 p 的每条代码路径上，表达式 e 都已经进行过求值，那么表达式 e 在位置 p 处是冗余的。

局部值编号（Local Value Numbering，LVN）

值得一提的是，我需要关注的一个优化方法叫做全局值编号（global value numbering）。这一节算是它的前置知识。

在介绍局部值编号的具体算法之前，我们显然要先了解一下“冗余”。考虑包含如下 4 条语句的基本程序块，我们将该块称为 B：

前面我们知道，对于一个表达式（比如 b + c），当且仅当它在 B 中此前已经计算过，且在此之间并无其他的运算重新定义组成表达式的各个参数值时，我们称它在 B 中是冗余的。在 B 中，第 3 个运算中出现的 b + c 不是冗余的，因为第 2 个运算重新定义了 b；第 4 个运算中出现的 a - d 是冗余的，因为在第 2 个和第 4 个运算之间 B 没有重新定义 a 或 d。

一般来说，将冗余的求值替换为对先前计算值的引用是有利可图的，但这种获利性是不能保证的。将 d <- a - d 替换为 d <- b 有可能会扩展 b 的生命周期并缩短 a 或 d 的生命周期，根据具体细节情况的不同，各个重写可能会改变对寄存器的需求。如果将冗余计算替换为引用会导致寄存器分配器逐出基本程序块中的某个值，那么这种重写很可能是无利可图的。但很明显，优化器没办法一致地预测寄存器分配器的行为，因为代码到达寄存器分配器之前至少还有可能会进一步变换，因此，用于删除冗余的大多数算法都直接假定，为避免冗余而进行重写是有利可图的。

生命周期：一个名字的生命周期是介于其定义位置和各个使用位置之间的代码区域。这里，定义意味着赋值。

人们已经开发出了许多用于发现并消除冗余的技术，局部值编号是这些变换中最老也最强大的技术之一。它可以发现基本程序块内部的冗余，并重写该程序块以避免冗余。它为其他局部优化（如常量合并和使用代数恒等式进行的化简）提供了一套简单且高效的框架。

Value numbering 顾名思义就是给各个变量和表达式进行编号，同时放入一个 map 中，后面有新的表达式，会优先在 map 中查找，如果有，则替换掉。没有，则插进去。

算法

直接地说，值编号的意义是指为每个 SSA 的 instruction 计算一个 Hash 值，然后遍历 instruction 寻找相同的 instruction 进行合并优化。

算法遍历基本程序块，并为程序块计算的每个值分配一个不同的编号。该算法会为值选择编号，使得给定两个表达式 和 ，当且仅当对表达式的所有可能的运算对象，都可以验证 和 具有相等的值时，二者具有相同的值编号。

下面我们用 LVN 指代局部值编号算法。

LVN 的输入是一个具有 n 个二元运算的基本程序块，每个运算形如 。LVN 算法会按顺序考察每个运算，它使用一个散列表来将名字、常数和表达式映射到不同的值编号。这个散列表会被初始化为空。

为了处理第 i 个运算，LVN 在散列表中查找 和 ，获取与二者相对应的值编号。如果算法能在散列表中找到它们对应的表项，那么 LVN 将使用该项包含的值编号；否则，算法将创建一个新的表项并为其分配一个新的值编号。

给出 和 的值编号，分别记作 和 ，LVN算法会基于 构造一个散列键，并在表中查找该键。如果存在对应的表项，那么该表达式是冗余的，可以将其替换为对此前计算值的引用（也即它对应的 值）；否则，运算 i 是改程序块中对此表达式的第一次计算，LVN 会为对应的散列键创建一个新的散列表项，并为该表项分配一个新的值编号。算法还会将散列键的值编号分配给对应于 的表项。因为 LVN 是使用值编号而非名字来构造表达式的散列键的，因此不这样没法通过赋值和复制操作来跟踪值的流动。

很显然，如果我们每个变量有唯一的编号，并且每个符号只定义一次，那么就非常方便我们做 LVN 了——这种形式其实就是SSA。具体的原因在 命名的作用 一节中再谈。

扩展 LVN 算法

LVN 提供了一个自然的框架，可用于进行其他几种局部优化。事实上，在我们的中端优化实践中，这三种局部优化不是可选项而是必选项，在进行正式的值编号之前，我们必须先使用这三种优化对输入的表达式进行处理，它们在 LLVM 中被总结为了 LLVM::InstructionSimplify 这个文件，可以在此处观看它的源码进行模仿学习：LLVM: lib/Analysis/InstructionSimplify.cpp Source File，可以视为一种体力活。（笔者已经写得快吐血了）

交换运算

对于可交换的运算来说，如果两个运算只是运算对象出现顺序不同（比如 a * b 和 b * a），那么二者应该分配相同的值编号。在 LVN 为当前运算的右侧表达式构造散列键时，它可以使用某种方便的方案对各个运算对象进行排序，比如按照值编号排序。这个简单的操作将会确保“同一”交换运算的不同变体分配到同一个值编号。

常量合并

首先来了解一个优化思路：常数折叠。它的动机是显然的。常数表达式计算（constant-expression evaluation），或称常数折叠（constant folding），指的是在编译时计算其操作数已知是常数的表达式。在多数情况下这是一种相对容易的转换。在它的最简单形式中，常数表达式计算包括判别表达式的所有操作数是否为常数值，在编译时计算此表达式，以及用计算结果替代该表达式。显然，至少对于布尔值，总是可以应用这种优化。

对于整型常数表达式，大多数情形下都可应用这种优化——但有些情形例外，即当执行这种常数表达式会导致运行时出现异常时，例如，零做除数，以及用其语义要求检测溢出的语言书写的常数表达式可能出现溢出时。在编译时对这种情形进行常数折叠需要判断对于程序可能有的输入，这些表达式在运行时是否实际会执行。如果会执行，可以用产生适当报错信息的代码来替代它们，或者（更可取的）在编译时产生警告信息指出可能发生的错误，或者同时产生这两种信息。对于地址表达式的特殊情形，常数折叠计算总是值得做并且是安全的——溢出与它们无关。

对于浮点常数表达式，情况要复杂一些。 （此处出过不少bug） 首先，我们必须保证编译时的浮点运算与目标机的一致，或者，如果不一致的话，编译器要提供适当的模拟器来模拟执行目标机的浮点运算。否则，编译时执行的浮点运算结果可能与运行时执行得到的结果不相同。其次，还存在浮点算术出现异常的问题，并且可能比整数的情况更为严重，因为ANSI/IEEE-754标准规定的异常和异常值类型多于已经实现的任何整数算术运算模式。可能的情形包括无穷值、NaN（非数值值）、非规格化的值，以及可能出现的（需要考虑的）各种异常。

同其他所有与数据流无关的优化一样，常数表达式计算的效果可以通过将它与数据流有关的优化（尤其是常数传播）结合在一起而得到增强。

常数折叠不仅仅适用于数字，字符串和常量字符串的连接也可以进行常数折叠。像 “abc” + “def” 这样的代码可以被替换为 “abcdef” 。但我们不涉及这个。

回到 LVN 框架。如果一个运算的所有运算对象都具有已知的常数值，那么 LVN 可以在编译时执行该运算并将结果直接合并到生成的代码中。LVN 可以在散列表中存储有关常数的信息，包括其值。如果 LVN 发现一个常量表达式，它可以将表达式替换为对相应结果的立即数加载操作。后续的复制合并（copy folding）会清理代码，消除不必要的复制操作。

代数恒等式

LVN 可以利用代数恒等式来简化代码。例如，x + 0 和 x 应该分配同样的值编号。遗憾的是，LVN 需要为每个恒等式增加特例处理代码，这需要一系列的条件判断（每个恒等式一个判断），而过多的条件判断语句会导致代码的运行速度大大降低。为改善这个问题，LVN 应该将这些条间判断组织到特定于运算符的决策树中。因为每个运算符只有少量的恒等式，这种方法可以使开销保持在一个比较低的水平。

显然，x + y 是冗余的，但对于第 4 条表达式，我们已经给 c 分配了值编号 3，但它不能被重写为 c <- a，因为 a 的编号已经不再是 3 了。此时我们就需要利用命名的性质去解决这个问题。（虽然我个人觉得在这里做一点 tradeoff 没什么不好……对手手.jpg）

一个方法是修改 LVN，使之维持一个从值编号到名字的映射，在对某个名字赋值时，比如 a，算法必须将 a 从其旧的值编号对应的列表中删除，并将其添加到新的值编号对应的列表中。这样，在进行替换时，算法可以使用当前对应于所述值编号的任何名字。这种方法显然是麻烦的，还会使得基本算法的代码变得杂乱。

另一种方法是编译器重写代码，为每个复制操作分配一个新的不同的名字，为每个名字添加一个下标来保持唯一性。这样可以使代码对每个值都有且只有一次定义，因而不会有值因重新定义而丢失。对于上面的例子，就是这样的情形：

间接赋值的影响

前面的讨论都假定赋值操作是直接且显然的，但许多程序包含了间接赋值，比如通过指针进行的赋值，或者对结构成员 / 数组元素进行的赋值，此时编译器可能就不知道需要修改哪个值或哪个内存位置了——因为它们导致编译器对值流动的推测出现了误差。

我们举指针赋值 *p <- 0 为例子，此时编译器缺乏对 p 可能指向的内存位置的具体知识，因此，编译器必须对该赋值操作可能修改的每个变量的下标都 + 1，这可能涉及所有变量的集合，但没有办法，因为这就是具有歧义的间接赋值对编译器能够推导出的事实集合的真实影响。编译器可以通过分析来消除指针引用的歧义，即缩小编译器认为指针能够访问的变量集合的范围，对于数组元素等等的访问也是如此。

歧义引用：如果编译器无法将某个引用“隔离”到一个单一的内存位置，那么该引用是具有歧义的。

超局部值编号（Super LVN）

只在单个 BB 块内执行 LVN 优化的话，可提供的时机是有限的。因此，一种扩展思路是将其扩展到 EBB 内。从之前的博文我们知道这已经来到区域优化的范畴了。假设一个函数有如下的 CFG：

可见，在 (A, B), (A, C, D), (A, C, E) 这几个 EBB 内可以做 value numbering ——对于它们而言，每个 BB 都有且只有一个前驱结点，说明这几个代码基本上是线性的，因此 A 的 value-number Map 还可以给 C 用，AC 的 value-number Map 对于 D 和 E 也依旧可以复用。为了实现复用，我们需要创建多余区域的 map，以便在不同的 BB 之间公用 map 的数据。

这种方法的难处在于如何使处理过程高效地进行。显然的方法是将每条路径当作一个 BB 来处理，遗憾的是，对于一个基本程序块来说，这种方法每处理一条包含该程序块的路径，都会分析该程序块一次，大大增加了编译时的成本。为此，超局部值编号算法通常利用 EBB 的树形结构。
