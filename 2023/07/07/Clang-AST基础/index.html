Clang AST 的基础结构

AST 中大部分的节点都是 Decl 类或者 Stmt 类的实例。前者即 Declaration ，表示声明，各种不同声明类型对应着 Decl 的各种子类（比如函数声明 FunctionDecl，参数变量声明 ParmVarDecl 等）；后者即 Statement，表示不同的语句，同样有很多子类，例如返回语句 ReturnStmt 、局部变量声明语句 DeclStmt 等，包括像 if 和 for 循环这样的结构也有对应的 Stmt 节点作为标识。AST 中还有许多节点不属于更大的层次结构，只能从特定的某些节点访问，例如 CXXBaseSpecifier 类型的实例。

需要关注 “被一对大括号包裹起来的是一个 code block” 这个概念。code block 的粒度和语句粒度结合，很大程度上决定了 AST 的结构。

注释语句不会被加入到 AST 中。

Decl class

一个 Decl 节点的结构如下图所示：

从编译原理知识可以知道，一棵 AST 代表的必然是一个函数 / 一个全局变量这样的 Global 型实体，此处讨论函数 AST 。

从根节点开始，每一棵函数 AST 的根节点必然是 FunctionDecl 的实例，FunctionDecl 可以拥有的子节点类型是 ParmVarDecl 的实例和函数体。函数体本身是 Stmt 的一个实例，例如 CompoundStmt （它表示一个代码块 code block ，即在 C 中用一对大括号包裹起来的代码部分）。

除了函数和函数参数，我们当然还可以对变量做出声明。不论是全局变量还是局部变量，其声明语句都对应一个 VarDecl 节点。如果该变量有初始值，那么该初始值会作为 VarDecl 的子节点出现在 AST 中，例如 int 型初始值会转化为一个 IntegerLiteral 节点（一种特殊的 表达式节点 ），节点的 value 域内容为 1 ，type 域内容为 int。

例如，当产生函数调用时，会生成 CallExpr 节点；产生隐式的类型转换时，会生成 ImplicitCastExpr 节点；引用已经声明过的变量或函数时，会生成 DeclRefExpr 节点；还有上文提到过的 IntegerLiteral 节点，等等。value 域内存储的可以是具体数值也可以是函数名这样的字符串。

由于 Stmt 节点本身的内容很单一，大部分信息都是通过它丰富的子节点（或者说以它为根节点的子树）提供的。例如：

  • 对于一个 CallExpr 节点，它的第一个子节点表征函数指针，后续子节点则是函数参数
  • 对于一个 IfStmt 节点，它有 4 个子节点，分别是：
    • VarDecl 类型 - 条件语句中可能存在的变量声明语句（在 c++ 中合法，c 中不行）
    • Expr 类型 - 条件表达式
    • Stmt 类型 - 具体代码块
    • Stmt 类型 - Else 块
  • 对于一个 ForStmt 节点，它有 5 个子节点，分别是：
    • Stmt - 循环计数变量的初始化语句
    • VarDecl - 可能存在的变量声明语句（在 c++ 中合法，c 中不行）
    • Expr 类型 - 条件表达式
    • Expr 类型 - 递增表达式
    • Stmt 类型 - 具体循环体

如何遍历 Clang AST

Clang 为用户访问 AST 提供了一个 Visitor 设计模式，具体而言，需要用到几个重要的功能类，分别是 CompilerInstance、ASTFrontendAction 、ParseAST、ASTConsumer 和 RecursiveASTVisitor ，一般来说加粗的部分需要用户重写 / 自定义一些接口函数。官方提供了详细的遍历工具书写教程：How to write RecursiveASTVisitor based ASTFrontendActions. — Clang 17.0.0git documentation (llvm.org)。

遍历的过程都已经被引擎自动实现了，我们需要做的只是调用合适的方法执行遍历，以及定义遍历过程中需要执行的具体操作。

进行任意操作之前，需要先创建 FrontendAction。FrontendAction 是一个接口，它允许 C 程序在编译过程中执行用户自定义的某些操作，因此所有的基于 Clang 编写的独立分析工具都需要基于它创建，对应到 AST 相关的操作是 ASTFrontendAction 这个子类。在 ASTFrontedAction 中需要创建 ASTConsumer —— 通过实现 CreateASTConsumer() 方法。该方法会对每个翻译单元返回一个 ASTConsumer 类型的对象，实现具体的遍历操作。

Clang AST 解析和构建的核心是 ParseAST() 这个接口，可以将文件解析成 AST 并通过钩子 ASTConsumer 把分析得到的 AST 节点传回给用户。这一步不需要用户重写或显式构造，由引擎自动完成解析，我们在 ASTFrontendAction 中重写 CreateASTConsumer 时，ASTFrontendAction 会帮用户调用 ParseAST() ，然后将自定义的 ASTConsumer 实例注册进去。

然后需要设计具体的 ASTConsumer 。设计 MyASTConsumer 的时候，就只需要遵循编译前端解析时的那些经验和自己的需求、重写 ASTConsumer 内部的一些钩子方法就可以了。

最重要的一个方法是回调函数 HandleTopLevelDecl() ，顾名思义，它会在我们访问每一个顶级声明（函数，全局变量）时被调用 —— 也就是说，需要将一系列函数和全局变量的声明作为函数参数引入，如下图所示。

对于每一个顶级声明 b ，我们需要调用一个方法 TraversDecl() ，它接受 Decl 类型变量作为函数参数，然后会自动 深度优先遍历 以该 Decl 节点为根的 AST 子树，对其中的每个节点调用类模板中对应的 visit 方法，如果该方法返回 false，就会停止递归遍历。

最后是 RecursiveASTVisitor 类，我们可以通过自定义这个类，重写其中的某些方法（例如 VIsitStmt()、VisitFunctionDecl() 等，其作用就是遍历到对应类型对象时被调用），决定我们的分析器在遇到某些类型的节点时的具体行为，Clang 提供了一系列丰富的 get 方法，调用它们可以获得节点的具体信息，比如行号、类型信息之类，我们可以在此处使用。RecursiveASTVisitor 为大多数类型的 AST 节点都提供了 bool 类型方法 VisitNodeType (NodeType *) 的钩子，但有一个例外是 TypeLoc 节点，因为它们是通过值传递的。

AST Context

翻译单元的 AST 的所有信息都捆绑在类 ASTContext 中。我们可以通过 getTranslationUnitDecl() 方法遍历整个翻译单元，或者访问经过解析的翻译单元的 Clang 符号表。

AST Matcher

直接使用 ClangAST 将语法树 dump 出来时，得到的结果是十分冗长的。此时我们自然地想到，可以用一种工具来过滤 dump 的输出。

这个工具就是 AST Matcher 。利用一个 Matcher 可以定义一些 “规则” ，用比较简单的方式构造一个过滤器，使得 clang 在 dump 语法树时只输出我们想要的一些内容，相当于是一个简化版的 RecursiveVisitor 。具体来说，ASTMatcher 提供了一个领域特定语言（domain specific language）来创建基于 Clang AST 的谓词（过程很像函数式编程），编写对应的程序（cpp）可以匹配我们想要的 AST 节点，并且能够通过访问节点的接口来获取它的属性、源位置等信息。

AST Matcher 是通过调用（嵌套的）构造函数来创建的。官网给了一个这样的例子：例如，要创建一个 Matcher 来匹配翻译单元 AST 中的所有类或联合声明，可以调用 recordDecl() 的 Matcher 构造函数。如果你还需要缩小匹配范围，例如为了找到所有名为 “Foo” 的类或联合声明，就可以插入一个嵌套的 hasName() Matcher，也即调用 recordDecl(hasName(“ Foo”)) 。你还可以使用多级内嵌 Matcher 进一步缩小匹配范围，例如，匹配从 “Bar” 类派生的所有类时可以调用 recordDecl(hasName(“Foo”), isDerivedFrom(“Bar”)) 。默认情况下，接受多级内嵌 Matcher 的 Matcher 会使用一个隐式的 allOf() 。

如何构造 Matcher

确定了自己需要过滤的节点类型之后，大致可以按照以下步骤构造自己的 AST Matcher：

  • 总结匹配模型
    • 在 Clang AST 中确定自己需要匹配节点的最外层类
    • 查找 AST Matcher Reference，查看是否有可以匹配自己需求的 Matcher ，或者可以进一步缩小的匹配范围，直到到达自己需要匹配的目标节点类型
      • 里面提供了许多的样例和节点的 AST ，基本上学习一下或者直接用都可以满足要求
  • 编写匹配表达式
    • 先针对最外层类创建自己的 外部 匹配表达式，并验证它有无按要求正常工作
    • 为自己想要匹配的内部节点构造内嵌 Matcher 并验证，然后逐步向下深入，直到目标节点被匹配（大致是一个下降的过程）

此时我们可以看到，关键是构造正确的匹配表达式。

匹配到感兴趣的节点后，我们还可以在其上执行一些其他的任务，例如构建源代码转换。所有的节点 Matcher 都是可绑定的，我们可以在符合匹配条件的节点上绑定一个字符串，稍后再匹配回调中这些内容会被返回，可以方便我们后续进行检索和调用等。例如 recordDecl(hasName(“MyClass”)).bind(“id”) 。
