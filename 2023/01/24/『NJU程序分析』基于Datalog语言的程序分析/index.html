『NJU程序分析』基于Datalog语言的程序分析

这篇博客对 Datalog 的讲解更加深刻，包括 Datalog 引擎 Souffle 的入门指南翻译：（三）Datalog和程序分析 | 技术和思考 (blog-blockchain.xyz)

Datalog 是一种声明式（declarative，最出名的声明式语言莫过于 SQL）的编程语言，在数据库和静态分析领域使用非常广泛。对比命令式（imperative）编程语言，声明式语言在一些特定场合下会更加有优势。

动机

其实很好理解，就是前面提到的所谓“特定场合”——静态分析的逻辑执行 / 判断过程。

如果使用 Java 之类的语言实现 SQL 或者说 Datalog 里面的一些功能，代码就会冗长重复，非常不方便，可读性也比较差。声明式语言的优势在于 抽象并屏蔽重复逻辑 ，偏向于描述工作规范而不是工作过程，因此它们的实现是精炼、易读、易于实现的（梦回数据库）。

Dalalog 简介

省流：分析的三板斧是 EDB、IDB（分析结果）和推导规则

Datalog 是一种由 Prolog 分化而来的逻辑编程语言，其高效实现之一是 Souffle。这里是 Souffle 的在线编译器：Souffle

它是作为一种数据库语言出现的（约莫 mid - 1980s。直到现在，在百度中搜索 Datalog，百科中依旧显示的是数据库语言的内容），现在，它已经有了许多不同的应用：

  • 程序分析
  • 声明式网络
    • 一种编程方法，使开发者能够简明地指定网络协议和服务，这些协议和服务会被直接编译到执行规范的数据流框架中
  • 大数据
  • 云计算
  • ……

Datalog = Data + Logic（与或非逻辑），其中 Data 是一些列的谓词，而 Logic 是很多规则，后面会讲解。它具有以下特点：

  • 没有副作用
  • 没有控制流
  • 没有函数
  • 不是图灵完备的
    • 因为其相比 Prolog 最大的不同就在于对语法做出了很多的限制，这些限制导致其非图灵完备

Data

Predicates：谓词

省流：谓词 是陈述的集合，可以理解成 SQL 中的表或者关系

在 Datalog 中，一个谓词（也称关系）是一组陈述，它是 Datalog 的一个主要组成部分。本质上，谓词是数据组成的一张（关系）表，表的每一行代表一个事实（fact）。一个事实断言了一个特定的元组（表中的一行）属于某一张关系表。也就是说，事实的含义就是：一个特定的值组合（该值组合一定是关系表中的一个元组）对于某个谓词来说是逻辑真的。

举个例子：在上图这张表中，Age 是一个谓词，表征了一些人的年龄信息。对于谓词 Age 来说：

  • (“Xiaoming”, 18) 表示 “Xiaoming is 18” 这个事实（它是真的）
  • (“Abao”, 23) 表示 “Abao is 23”，它不是一个事实

Atoms：原子

原子是 Datalog 中的基本元素，包括关系原子和算术原子。

形如 的原子称为关系原子（relational atom），当谓词 P 包含元组 时取值为真。

关系原子中的 Terms 可以是具有任何值的变量，也可以是常量。

以上节中的谓词 Age 为例，下面这些都是关系原子：

  • Age(person, age)
  • Age(“Xiaoming”, 18)

除了关系原子，Datalog 中还有算术原子（arithmetic atom），格式形如 attr >= var，其中 attr 是谓词中的一个列，而 var 是一个变量或者常量。

Logic

逻辑的表达称为规则（rules），形式为 H <- B1, B2, ..., Bn。其中：

  • H：规则的头部，是规则的结论（consequent）
  • B：规则体，是规则的前因（antecedent）
    • 被称为子目标（subgoal）
    • 当且仅当前因为真时，结论为真

这很容易让我们联想到之前提到过的 推导式 。

在 Datalog 的规则中，“,” 等价于逻辑与，“;” 等价于逻辑或，而 “!” 依旧是取反的意思。例如，假如我们要在前面提到的 Age 表中选取年龄大于 18 岁的人名，就可以构造这样一条规则：Required(person) <- Age(person, age), age >= 18。

想要实现逻辑或，还可以通过书写多条具有相同头部的规则实现。“;” 的优先级要低于 “,”，因此析取关系可以用圆括号括起来，like：H <- A, (B; C).

如果没有递归结构，那么 Datalog 就只能支持基本关系代数的查询了。当然递归还引出了两个个要求，就是不能出现 无穷 推导和 矛盾 推导。因此 Datalog 首先为避免无穷推导定义了一个概念叫做 规则安全 （rule safety）——其定义是，一条规则是安全的，当且仅当 规则体中的每一个变量至少出现在一个非取反的关系原子中 （很好理解，必须对变量取值做有限约束）。Datalog 中只允许出现安全规则。

例如，如下的两条规则都是不安全的：

  • A(x) <- B(y), x > y
    • x 只出现在算术原子里
  • A(x) <- B(y), !C(x, y)
    • x 只出现在取反的关系原子里

至于矛盾推导，为了避免前因后果发生矛盾，Datalog 禁止一个关系的取反和递归同时出现在一条规则中。比如 A(x) <- B(x), !A(x)。

理解 Datalog 中的 Rule：枚举所有前因中各个项的谓词表中的记录，然后对它们做笛卡尔积组合，选择那些逻辑运算结果为真的组合，用它们对应的结论形成最后头部原子的结论——也即，规则头部（which is a predicate）是由所有的逻辑真原子组成的。

Datalog program = Facts + Rules

外部数据库和内生数据库（EDB & IDB）

Datalog 中的谓词被分为了两种类型：

  • 外部数据库（Extensional Database，EDB）
    • 在 前因 中被定义的那些谓词
    • 其关系是不可改变的
    • 可以视为输入关系
  • 内生数据库（Intensional Database，IDB）
    • 只能被规则推导出来的谓词，也即 结论
    • 其关系是通过规则推断出来的
    • 可以视为输出关系

观察一段规则 H <- B1, B2, ..., Bn，其中 H 只可能是 IDB，而 既有可能是 EDB 也有可能是 IDB。

Datalog 程序的执行

Datalog 引擎通过给定的规则和 EDB 谓词来推导事实，直到没有新的事实可以被推导出来。也就是说，推导规则时，程序会一直运行到 IDB 不再发生变化时才终止。

一些现代 Datalog 引擎：LogicBlox, Soufflé, XSB, Datomic, Flora-2, …

Datalog 程序具有两大重要特性：

  • 单调性
    • 因为事实不能被删除
  • 程序（or 推导）必然终止
    • 因为单调性保证了事实是有限的
    • 规则安全保证了 IDB 谓词的可能取值也是有限的

Datalog 分析的优势基本上是由其声明式特性带来的，即简洁易读、容易实现，而且容易受益于执行引擎的优化。但其劣势也是突出的，作为非通用编程语言，它的表达能力很有限，而且也很难实现自定义优化。
