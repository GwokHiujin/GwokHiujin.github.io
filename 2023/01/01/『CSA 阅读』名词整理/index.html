静态分析：名词杂项

此处统一记录一些在静态分析学习过程中遇到的名词杂项，包括一些非常简短的网课 / 会议 ppt / 博客 的学习笔记。

操作语义：operational semantics

总结：操作语义是编程语言执行的数学模型，具体而言，是单步转移判断的自反传递闭包。本质上，它是一个数学定义的解释器。

什么是语义

语法和语义构成了程序语言的两个基本组成方面，前者表征程序的结构，后者解读程序的意义。也就是说，语义需要解决的问题就是——这个程序的意义是什么。

那么我们该如何定义一个程序的意义呢？有以下这些正式和不正式的 规范 （相当于从不同的角度阐释一个程序的意义） 供我们参考：

  • 正式规范

    • 指称语义（denotational semantics）：将 terms 直接关联到 values
    • 操作语义（operational semantics）：描述如何对 term 求值的过程
    • 公理化语义（axiomatic semantics）：描述对一个 term 求值会产生的影响
    • ……

  • 非正式规范

    • 参考实现（reference implementation）：在某些具体实现下执行或者编译程序会产生的结果
    • 程序员的直觉（community / designer intuition）：人们认为一个程序应当如何执行才算是正常执行

显然，虽然我们总是在用非正式的规范衡量自己写出来的程序的意义，但一个正式的规范是更加有助于我们理解程序运行及其结果的。 形式语义 就是这样一个我们需要的“规范”。形式语义至少拥有以下好处：

  • 比具体实现更简洁，比人脑直觉更精确

    • 能够回答：一个具体实现是否正确

  • 支持分析和转换的定义

    • 能够证明该语言的属性
    • 能够证明用该语言编写的程序的属性

  • 促进更好的语言设计

    • 帮助程序员更好地理解设计决策的影响
    • 应用语义的洞察力来改进语言设计（例如组合性）

什么是操作语义

操作语义的含义是：通过描述一个程序中的变量是如何被求值的，来定义一个程序的“意义”（meaning）。

操作语义有自然语义（natural semantics，又称 big-step semantics）和结构操作语义（structural operational semantics，又称 small-step semantics）两种形式。前者定义转换关系 ，表示对最终状态的计算，在这种风格下，语义就是这种转换关系本身；后者则定义了转换关系 ，表示求值的一个步骤，在这种风格下，语义是这个关系的自反传递闭包 。

符号执行

总结：符号执行是一种用抽象的符号表示程序中变量的值，来模拟程序的执行的静态分析方法。

符号执行（Symbolic execution，也称为符号求值）是一种分析程序以确定何种输入会导致程序的每个部分执行的方法。求值的具体过程是：一个跟随程序的解释器会假设输入为符号值，而不是像正常执行程序那样获得实际输入。接着，解释器根据程序中的表达式和变量符号等得到表达式，并根据每个条件分支的可能结果的符号得到约束。表达式 + 约束就得到了执行的符号结果。

套用以前记的笔记：

符号执行用抽象的符号表示程序中变量的值，来模拟程序的执行。由于它跟踪了变量的所有可能取值，因此能够发现程序中细微的逻辑错误。但是在处理大程序时，程序执行的可能路径数随着程序尺寸的增大而呈指数级增长。在这种情况下需要对路径进行选择，选取一定数量的路径进行分析。

  • 克服了在静态测试时不能确定程序中变量的值的问题

  • 常在对路径敏感的程序分析中使用

  • 限制

    • 符号化地执行所有路径的方式很难扩展到大型程序上

    • 可能遇到死循环

    • 存在路径爆炸的风险

    • path-by-path 的分析方式相较于 input-by-input 的动态分析方式的优势不稳定，具有一定的程序依赖性，比如一个程序的 inputs 可能很集中，有 99% 的时间都跑在特定路径上，这个时候 path-by-path 的分析效率就很低了

    • 很难静态地识别出别名，对于一些严重的情况，例如某个 memory 被外部修改了，而我们对 memory 一无所知，那么此时就只能 invalidate 所有的 memory location

    • 环境交互问题（Environment Interactions）：显然这是所有静态分析工具都有的限制

      • 解决办法：对环境直接执行调用；对环境建模；fork 整个系统状态

缓解路径爆炸的方法一般有：启发式地优先执行特定路径 / 并行执行 / 合并相同路径。CSA 目前只采用了最后一种方式。

符号执行与抽象解释

总结是没什么太大关系，可能我之前听李樾谭添老师课程里的“Abstraction”，把我给小小地误导了一下。不过它们也有思想相通的地方，比如 joining 和 widening 的分析思想，包括分析过程中使用抽象状态这一点也很相似。

抽象解释是一种计算机程序语义的合理近似理论，它基于有序集上的单调函数，尤其是 格 。它可以被看作是计算机程序的部分执行，在不执行所有计算的情况下获得有关其语义的信息。

程序语义通常在循环或递归过程中使用固定点来描述。我粗浅的理解就是，使用抽象解释函数作用于一个特定的抽象域，只考察程序的某一特定属性，在抽象域上找程序的最小不动点。

显然，数据流分析就是抽象解释的一种应用。

Soundness，Soundiness & Unsoundness

首先回顾一下 sound 的定义： 不漏报 程序中的 bug 的，或者说能够将程序中所有可能的执行建模的分析，就称为一个 sound 的分析，这种“不出错”的特性叫做 soundness 。

然而，无论是在学界还是工业界的实践中，我们都无法完成一个真正 sound 的分析——几乎所有已发表的全程序分析在应用于真正的编程语言时都是不健全的。其原因是，针对很多编程语言拥有的高级特性（称为 Hard Language Feature），我们为了防止发生误报（即为了保证分析的安全），只能够做一些积极保守的 tradeoff ，导致最后的分析算法是 unsound 甚至非常不精确的。

比如对 C/C++ 中的一个指针 p ，可以加上某个经过运算的偏移量 x ，在分析中，为了保证安全，我们只能保守地假设 p+x 可以指向内存中的任何地方。

Soundiness 这个概念是相对 Soundness 的理想性提出的，也即它针对的是现实情况：一个 Soundy 的分析方法是近乎 Sound 的，同时针对一些它无法正确处理的语言高级特性，该方法应当给出无法处理这些特性的具体原因。
