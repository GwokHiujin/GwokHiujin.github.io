『NJU程序分析』静态分析与代码安全

关注的主要是在今年所公布的网络安全议题中占据高位的两个问题：

  • Injection errors
  • Information leaks

更像是一节轻松的科普课，学习重点应当在污点分析及其作业上，算法跟指针分析是一样的。

信息流安全

在静态分析概论处我们已经强调过，静态分析很大一部分的用武之地在信息安全领域上，其中一部分是信息流安全（Information Flow Security）。动机很显然，就是防止我们不希望其产生的信息流产生（比如骇客非法获取敏感信息），以保证信息安全。

信息流安全将信息流分析与网络安全桥接了起来。其基本策略是：

  • 为变量定义 安全等级
    • 这一点可以类比 Linux 中的用户权限管理，如 root 用户和普通用户
  • 设定不同安全等级之间的信息流协议（Information Flow Policy）
    • 比如 Linux 中的 root 用户可以做一切事情，而普通用户不可能访问 root 用户专有的文件和文件夹

最常用的是两阶段协议（Two-Level Policy），关于其具体定义等内容请自行复习数据库课程的相关知识。此处需要注意的是，我们可以将安全领域中密级（Security levels）这个概念建模成静态分析中的 格 （Lattice）。通过这种方式，就可以连通信息流分析与网络安全。

Dorothy E. Denning, “A Lattice Model of Secure Information Flow”. CACM 1976.

secure-information-flow.pdf (washington.edu)

还有一种常用的策略是 Noninterference policy，其含义是高密级的信息不应该影响到低密级的信息，这能够保证攻击者无法通过观测低密级的信息反推出高密级的信息。如果从格的角度描述，其含义就是保证信息流只能在格中向上流动（L to H）。

访问控制 vs 信息流安全

“A practical system needs both access and flow control to satisfy all security requirements.”

–D. Denning, 1976

访问控制是一种用于保护敏感信息的标准方法。它的原理是检查程序是否有能够访问特定信息的权限或者许可，关注信息是如何被访问的。

而信息流安全更像是一种端到端的方法。它追踪信息是如何在程序中流动的，以确保程序能够安全地处理信息，关注信息是如何被传播的（访问之后发生的事）。

  • 信息流

若变量 x 中的信息被传播到变量 y，那么就生成了一条从 x 到 y 的信息流，表述为 x -> y。

（看起来有点像指针分析，而它们也确实有点联系）

保密性与完整性

  • 保密性（Confidentiality）：侧重于防止秘密信息的泄露，即保护关键数据不被攻击者读取
    • 方向是从秘密信息到攻击者
  • 完整性（Integrity）：侧重于防止不受信任的信息破坏（受信任的）关键信息，即防止攻击者通过恶意提权或者 SQL 注入等手段执行高执行权限的命令，写入关键数据
    • 方向是从攻击者到秘密信息
    • 可能的问题: Injection errors （2013-2019 年漏洞的第一大原因）
      • 命令注入
      • SQL 注入
      • XSS 攻击
      • ……
    • 这个定义是在信息流安全语境下的，在其他语境（比如我们在数据库课堂上接触到的广义语境）有更多其他的含义，此处不赘述

保密性与完整性是一对二元关系，一个保护读，一个保护写。从密级的角度考虑，保密性讨论的是数据从高密级处（Secret）流动到低密级处（Public）的可能性，也即信息流在格中从 L 流动到 H 的错误；而完整性讨论的则是数据从低密级处（Untrusted）流动到高密级处（Trusted）的可能性，也即信息流在格中从 H 流动到 L 处这种 general 场景下发生的问题。

  • Timing channels: 计时信道

  • Exceptions: 异常情况

    • 通过程序抛出的异常来传递信号信息

不过通常来说，显式流携带的信息比隐蔽信道更多，所以我们重点关注前者。

污点分析：Taint Analysis

前面提到，为了保证信息流安全，我们需要防止不希望其产生的信息流产生。那么静态分析如何应用于其中呢？我们知道静态分析是在程序运行之前执行的，因此静态分析的用武之地就是分析程序的信息流，进而找到程序中是否有可能产生非法信息流的罅隙。

污点分析是最常用的一种信息流分析。它将程序中的数据分为两类：

  • 我们关注的，或者说感兴趣的数据。我们为这些数据附上一些标签，将其称为污点数据（tainted data）
  • 其他数据，称为非污点数据（untainted data）

污点数据源被称为 sources 。在实践中，污点数据通常来自某些方法的返回值，因此我们将这些方法视为 sources。污点分析跟踪污点数据在程序中的流动，并观察它们是否会流动到我们所关心的一些位置（称为 sinks ）。在实践中，sinks 通常是一些敏感的方法。

Source --tainted data–> Sink

将其类比到前面提到的保密性与完整性中：

  • 保密性

    • Source：秘密数据的数据源

    • Sink：数据的泄露点

    • 若有关注的信息流：发生信息泄露

  • 完整性

    • Source：不受信任的数据源

    • Sink：关键的计算

    • 若有关注的信息流：发生 Injection errors

污点分析可以探测到这两种我们不想要的信息流。

污点分析与指针分析

将“污点数据是否能从 source 流动到 sink”的命题转换一下，就可以叙述成“sink 处的一个指针能够指向哪些污点数据”。显然，结合指针分析可以办到这一点。它们的本质都是追踪数据在程序中的流动。

因此我们只需要：

  • 将污点数据当成指针分析中的人造对象
  • 将 sources 当成指针分析中的分配点（污点数据的分配点）
    • 也即 sources 就是源方法的集合，对源方法的调用将返回污点数据
  • 利用指针分析来传播污点数据，观察是否能到达 sink
    • sinks 此时被处理为一组具有我们关心的敏感参数的 sink 方法，流向这些方法的参数的污点数据被认为违反了安全协议

分析的算法与指针分析是相同的。

一般来说，污点分析的输出是一组形如 <i, j, k> 的元组数据，它的涵义是来自调用点 i 处的污点数据可能会流向调用点 j 处的第 k 个参数。
