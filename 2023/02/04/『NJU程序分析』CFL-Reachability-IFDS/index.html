『NJU程序分析』CFL-Reachability & IFDS

IFDS 框架本身的概念比较抽象，理论也比较杂，需要好好理解。讲课的主要动机是回顾之前的分析理论与相似的概念，具体的还得从实践中去体会（写 checker 写得已崩溃）

本质上是通过 图可达 这个概念去表述程序分析问题。重点是了解爆炸图和理解 Tabulation 算法。

Feasible Path & Realizable Path

为了定义 CFL-Reachability 引入的基本概念。

在实际项目中，真实程序的控制流图往往是非常复杂的。只不过我们也知道，控制流图中并非所有的路径都会被执行到，因此，我们希望在进行程序分析时只分析可能被执行的路径（称为 feasible paths），或者说，让那些不可行（infeasible）路径尽可能少地“染指”我们的分析过程。不可执行的假边越少，分析的误报就越少。

然而，一条路径本身是否可行（feasible）是一个不可判定（undecidable）问题。比如，有些路径的可行与否跟程序语义相关，只能在动态执行的时候被发现，那么静态分析就没法判断其可行性了。

此处定义一个新概念 Realizable Path：Realizable Paths 指的是那些返回与其相应的调用位置相匹配的路径，有点像括号匹配。引入这个概念后，我们分析的目标就变成了识别 Realizable Paths，这样我们就可以避免那些 unrealizable paths 污染分析结果——因为 executable / feasible path 实际上可以看作是 realizable path 的真子集。

可见这样可以排除 {18/30, -1} 两个假边引入的 false positive。

解决这个匹配问题的思路跟括号匹配是一样的——学过编译理论后我们知道，使用 上下文无关文法 就可以识别一个匹配的括号串。而这就是下面要介绍的方法 CFL-Reachability。

IFDS 概述

IFDS 是一种基于 图可达性 的程序分析框架，提供精确的过程间数据流分析。

IFDS 是四个描述其分析特性的单词的首字母组合：Interprocedural、Finite、Distributive、Subset Problem，意思是只有当你的分析问题是全过程的、 具有 distributive flow functions over finite domains 的数据流分析时，才可以使用 IFDS 框架进行分析。

其特性核心是 finite 和 distributive 。distributive 的含义是 。

IFDS 提出的是一种 MRP（Meet-over-all-Realizable-Paths，也即为 MOP 思路加上一个条件：限制 Meet 操作的对象为 Realizable Paths）的分析思路。MRP 相对于 MOP 提供的是一个更精准的结果。

有关 Meet、MOP 等概念的理论基础可以在 『NJU程序分析』数据流分析理论基础(FD) 中回顾。

抽象框架

IFDS 本身没有一个特别清晰的结构。其框架可以大致抽象如下：

  • 输入：给定一个程序 P 和一个数据流分析问题 Q

  • 为 P 构造一张 supergraph，称为 G*，并基于 Q 为 G* 中的边定义 flow functions

    • supergraph： G* = (N*, E*) 。这个星号表征的是 super 特性，也即 G* 是程序中 每一个过程 的流图组成的集合（很显然每一个方法内部都会有自己的执行过程，而每一次调用也是过程，会产生边）。

      对于 G* 中的每一个子图 ，它都有一个独一无二的入口点 和出口点 。而每一个调用过程都会被一个调用点 和返回位置点 表征。

      因此，我们可以将 supergraph 理解成 IFDS 分析框架下的控制流图（过程间 + 过程内）。其中需要了解的是以下三种边：

      • [过程内]从 到 的过程间 call-to-return-site 边
        • 允许传播过程内的 local 信息，可以提高分析的精度，减少 false positive（不影响分析的 soundness）
      • [过程间]从 到被调用者入口位置 的 call-to-start 边
      • [过程间]从被调用者出口位置 到 的 exit-to-return-site 边

  • 通过将 flow functions 转化为表示关系（图），为 P 构造出一张 exploded supergraph，称为 G#

    • 也即根据 flow functions 和输入在 supergraph 中构造出许多的小子图，具体的构造过程在下文中介绍。

  • 通过对 G# 施加 Tabulation 算法找 Realizable Paths，Q 可以作为一个图可达问题被解决（即可以找到 MRP solutions）

假设 n 是一个程序点，IFDS 分析的结果就是：data fact d ，当且仅当在 G# 中存在一条从 <, 0> 到 <n, d> 的 realizable path。

什么时候使用 IFDS

本质是理解 IFDS 的 Distributive 特性。我们前面提到，其含义是： ，IFDS 中的每个流方程一次只能处理一个 input data fact，因此：

  • 如果我们需要考虑多重的 input data facts 以产生正确的输出，那么这个分析就不是 distributive 的，不应该使用 IFDS 框架分析
    • 比如常量传播，给定一个 z = x + y 语句，显然我们不能在仅仅知道 x 或者 y 的值的情况下决定流方程的输出
    • 比如指针分析，别名信息就是一种多重输入，而没有别名信息我们是无法递归地进行指针分析的
